核心思路

  你要的本质是一个「本地 Web 终端管理器」——类似 VS Code 内置终端 +
  项目分类面板。业界成熟做法是：

  浏览器 (Firefox/安卓/iPad...)          Windows 主机
  ┌─────────────────────────┐      ┌──────────────────────────┐
  │  项目/任务分类面板        │ HTTP │  Node.js 服务器           │
  │  xterm.js 终端渲染       │◄────►│  node-pty 伪终端          │
  │  (WebSocket 实时收发)    │  WS  │  → powershell/claude/codex│
  └─────────────────────────┘      └──────────────────────────┘

  - 后端：Node.js + node-pty（Windows 上走 ConPTY），每个 task
  对应一个伪终端会话，在指定目录启动 powershell，再自动输入 claude 或 codex
  - 前端：xterm.js 渲染终端（VS Code 同款，Firefox/移动端浏览器全兼容），WebSocket
  双向传输
  - 数据：项目/任务结构存一个本地 JSON 文件即可，无需数据库
  - 跨设备：服务器跑在 Windows 上，监听局域网地址，安卓/iPad 用浏览器访问
  http://192.168.x.x:端口 即可——不需要互联网，符合你"仅本地"的要求

  一个需要你注意的关键取舍

  你提到「powershell 先同步打开方便查看效果」——即又开真实 PowerShell
  窗口、又在网页镜像。这两者技术上是冲突的：

  - node-pty 方式（推荐）：进程挂在伪终端上，网页就是那个窗口，全彩色、可交互、可杀进程
  。但桌面上不会弹出真实的 PowerShell 窗口。
  - 真实窗口 + 镜像方式：用 Start-Process 弹真窗口，网页只能靠 Start-Transcript
  日志文件 tail 出只读、无颜色、有延迟的内容，且网页无法向它输入。

  我的建议：直接采用方式一，把网页当作终端本身（这正是你"快速找到窗口"的最终形态）。开
  发阶段想对照效果，可以临时并排开一个手动的 PowerShell
  跑同样命令对比，而不是做双写镜像。提示词里我按方式一写，并注明了这个决策。

  Ctrl+G 输入

  在网页里实现成：终端聚焦时按 Ctrl+G → 弹出大文本框（modal），写完点"发送并关闭"→
  内容整体写入终端 stdin。比调用系统记事本更简单且全平台通用。

  分阶段

  1. 骨架：服务器 + 项目/任务 CRUD + JSON 持久化 + 左侧分类树 UI
  2. 终端核心：node-pty 会话、xterm.js 渲染、WebSocket 流、指定目录启动 claude/codex
  3. 管理功能：Ctrl+G 输入框、杀进程、断线重连/回滚缓冲（关网页再打开内容还在）
  4. 打磨：局域网访问、移动端触屏适配、会话状态标识（运行中/已退出）
